# Техническое задание

## Название проекта

Чат-бот для `MAX` на базе `DeepSeek`

## Цель

Разработать готовый к деплою сервис, который:

1. Принимает текстовый запрос пользователя в `MAX`.
2. Отправляет запрос в `DeepSeek`.
3. Возвращает пользователю текстовый ответ.
4. Сохраняет информацию о пользователях, промптах, ответах и вызовах модели.
5. Предоставляет внутреннюю web-панель для просмотра статистики, истории диалогов и ошибок.

## Исходные требования

Минимальные требования по задаче:

1. Используем `DeepSeek`.
2. По промпту юзера бот выдает ответ.
3. Бот собирает информацию о юзерах, промптах и ответах.

Дополнительное требование:

- сделать проект под ключ
- оформить его как два отдельных Python-репозитория
- подготовить базы, инфраструктуру и документацию

## Структура проекта

Проект разделен на два Python-репозитория:

- `max_chat_backend`
- `max_chat_frontend`

### Backend

Назначение:

- webhook и polling-интеграция с `MAX`
- обработка входящих сообщений
- работа с plain text из `message.body.text`
- сохранение пользователей, диалогов, сообщений и ответов
- вызовы `DeepSeek API`
- очередь задач и retry
- внутренний API для админки

Стек:

- `FastAPI`
- `PostgreSQL`
- `Redis`
- `Celery`
- `SQLAlchemy`
- `Alembic`
- `httpx`

### Frontend

Назначение:

- внутренняя админка для команды
- просмотр пользователей
- просмотр диалогов и сообщений
- просмотр ответов модели и ошибок
- CSV-экспорт

Стек:

- `FastAPI`
- `Jinja2`
- `server-rendered HTML`

## Функциональные требования

### Пользовательская логика

- команда `/start`
- команда `/help`
- команда `/new` для сброса контекста
- прием обычного текстового сообщения
- ответ от `DeepSeek`
- поддержка многошагового диалога
- callback `Новый диалог`

### Работа с текстом

- входящий текст берется из `message.body.text`
- raw webhook payload хранится только для аудита
- бизнес-логика не строится на raw-представлении сообщения
- длина пользовательского текста проверяется по plain text
- длинные ответы режутся на несколько сообщений под лимит `MAX`

### Сбор данных

Сохраняются:

- пользователи
- диалоги
- пользовательские сообщения
- ответы ассистента
- вызовы `DeepSeek`
- webhook-события
- ошибки обработки

### Защита и устойчивость

- rate limiting
- ограничение количества активных запросов на пользователя
- blocklist-модерация
- retry на внешние вызовы
- readiness healthcheck
- структурированные логи

## Нейросеть

Базовый провайдер:

- `DeepSeek API`

Базовая модель для `v1`:

- `deepseek-chat`

Архитектура построена через provider adapter, чтобы при необходимости можно было заменить модель или провайдера без переписывания бизнес-логики.

## База данных

Используется `PostgreSQL`.

Схема создается миграциями `Alembic`, вручную таблицы создавать не нужно.

Основные таблицы:

- `users`
- `conversations`
- `messages`
- `llm_requests`
- `webhook_events`
